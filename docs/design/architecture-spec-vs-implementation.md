# 全节点式 AI 视频创作：设计规格与当前实现对照表

本文档将产品层「DAG 驱动的模块化影视流水线」设计逻辑，与 **vibevideo** 仓库当前实现做逐项对照，便于迭代时对齐语义、避免仅停留在 UI 层讨论。

**对照依据代码路径（截至文档编写时的结构）：**

- 前端：`src/store/projectStore.ts`、`src/lib/types.ts`、`src/lib/serialization.ts`、`src/lib/flowEdge.ts`、`src/components/FlowCanvas.tsx` 等  
- 后端：`src-tauri/src/graph.rs`、`src-tauri/src/executor.rs`、`src-tauri/src/db.rs`、`src-tauri/src/lib.rs`

**图例：**


| 标记  | 含义                |
| --- | ----------------- |
| ✅   | 已有较明确实现，与设计方向一致   |
| ⚠️  | 有部分实现，但未到设计描述的完整度 |
| ❌   | 基本未实现或仅 UI/占位     |

## 一、全节点通用底层（设计文档第一章）

### 1. 节点「四件套」抽象（输入端口 / 算子内核 / 输出端口 / 参数控制面板）


| 设计要求              | 当前实现                                                                                                 | 对照  |
| ----------------- | ---------------------------------------------------------------------------------------------------- | --- |
| 多端口、多类型、协议校验      | React Flow 上多为统一 `in` / `out` Handle；`FlowNodeData` 为宽松结构（`prompt` / `path` / `video` 等），**无端口类型系统** | ❌   |
| 算子内核为唯一改数据单元      | 文生图等可走独立 `invoke`；`execute_graph` 内对 LLM / 媒体 / FFmpeg 有串行算子，与画布上单节点生成等 **两条路径并存**                   | ⚠️  |
| 参数面板与算子 schema 绑定 | 各节点面板直接改 `data` 字段；与 Rust 执行器之间 **无统一参数 schema**                                                     | ⚠️  |


### 2. DAG 拓扑执行引擎


| 设计要求                            | 当前实现                                                           | 对照  |
| ------------------------------- | -------------------------------------------------------------- | --- |
| 拓扑排序                            | `graph.rs`：`topological_order` + `validate_dag`（检测环路并拒绝执行）     | ✅   |
| 依赖校验                            | 校验边端点对应节点存在；**无「端口类型是否允许连接」校验**                                | ⚠️  |
| 节点状态机（未执行 / 等待 / 执行中 / 成功 / 失败） | `runs.db` 记录 run 级状态；**单节点状态未作为一等模型暴露给前端**；失败会中断整次 `run_graph` | ⚠️  |
| 失败阻断下游、断点续跑                     | 单节点失败导致整次运行失败；**无「仅重跑失败子图」**                                   | ❌   |
| 无依赖链路并行调度                       | `executor.rs` 中 **严格按拓扑顺序串行 for 循环**                           | ❌   |


### 3. 统一数据协议（资产 ID、元数据、格式适配）


| 设计要求                  | 当前实现                                                                                       | 对照  |
| --------------------- | ------------------------------------------------------------------------------------------ | --- |
| 全局资产 ID，节点间只传 ID      | **M1+**：`runs.db.assets` 有 `asset_id`；执行器 `resolve_node_media_rel_path` 优先 `assetId` 查库再回退 `path`（`executor/asset_resolve.rs`） | ⚠️  |
| 标准化元数据（分辨率、镜头 ID、时序等） | **M1**：图片/音视频 `meta_json` 含基础宽高或时长等；**分镜 / 时序未形成全图统一坐标系**                                  | ⚠️  |
| 数据格式自动适配算子            | **未见**通用适配层                                                                                | ❌   |


### 4. 全局资源与状态管理


| 设计要求          | 当前实现                                                  | 对照  |
| ------------- | ----------------------------------------------------- | --- |
| 项目持久化         | `canvasflow.json` + `serializeCanvas` / `parseCanvas` | ✅   |
| 撤销重做          | Zustand 图快照 `undo` / `redo`，带防抖合并                     | ✅   |
| 缓存分层与清理       | 工程目录素材与运行库；**未见**与设计文档同级描述的缓存策略                       | ⚠️  |
| 算力调度（本地 / 云端） | 设置中 Provider；**无统一调度器抽象**                             | ⚠️  |


### 5. 实时渲染与预览


| 设计要求                   | 当前实现                             | 对照  |
| ---------------------- | -------------------------------- | --- |
| WebGL/Canvas 全片预览、增量渲染 | 以节点内媒体预览为主；**未见**时间轴级全片预览与增量渲染管线 | ❌   |

## 二、各类型节点（设计文档第二章）— 摘要对照


| 设计中的节点/能力                            | 当前实现要点                                                                                        | 对照  |
| ------------------------------------ | --------------------------------------------------------------------------------------------- | --- |
| **文本节点**：Prompt 结构化、权重、下游联动重置        | `prompt` + `params`；无独立结构化解析引擎；联动依赖前端改数据                                                      | ⚠️  |
| **脚本节点**：分镜解析、全局时间轴、镜头 ID、一致性注入、批量映射 | `scriptBeats` / `storyboardShots` 可持久化；无自动剧本解析、无全局时间轴引擎、无对下游的自动节点绑定                           | ⚠️  |
| **图片节点**：文生图、特征向量、下游强制参考             | `generate_image_asset` 等；无独立视觉特征向量协议与引擎级「视频锚定」                                                | ⚠️  |
| **图片处理节点**（高清/修图/风格化等）               | **无**独立节点类型                                                                                   | ❌   |
| **视频生成**：帧一致、时长对齐脚本、插帧等              | `videoNode` + `video` 草稿与大量前端 UI；`execute_graph` 路径上 **未**作为完整视频推理算子跑通（多数字段为 skip 或旁路 invoke） | ⚠️  |
| **运镜节点**                             | **无**                                                                                         | ❌   |
| **视频编辑节点**（裁切/调速等，非简单 concat）        | **无**                                                                                         | ❌   |
| **TTS**                              | `AudioTtsPanel` 等 **占位**                                                                      | ❌   |
| **音频编辑/混音**                          | **无**                                                                                         | ❌   |
| **视频合成**：多轨、脚本时间轴对齐、转场               | `ffmpegConcat` 按 `inputs` 等拼接；**无**多轨时间轴合成引擎                                                  | ⚠️  |
| **字幕节点**                             | **无**                                                                                         | ❌   |
| **导出节点**                             | 导出路径分散；**无**独立导出算子节点抽象                                                                        | ❌   |

## 三、全链路数据流（设计文档第三章）


| 阶段（设计）         | 当前实现                          | 对照  |
| -------------- | ----------------------------- | --- |
| 语义输入 → 脚本结构化分镜 | 脚本内容以用户/表格编辑为主；**无**自动分镜管线    | ⚠️  |
| 关键帧 → 特征 → 视频  | 前端多依赖引用图/路径；**无**特征向量链路       | ⚠️  |
| 音画按镜头 ID 对齐    | **无**全局镜头 ID 驱动时间轴            | ❌   |
| 合成汇聚           | 仅 concat 等局部能力                | ⚠️  |
| 成片导出           | 工程内路径 + FFmpeg；**未**抽象为统一导出节点 | ⚠️  |

## 五、技术栈映射（设计文档第五章）


| 设计中的层次       | 当前技术实现                                                  | 对照                        |
| ------------ | ------------------------------------------------------- | ------------------------- |
| 视图层（画布/参数面板） | React + React Flow + 节点组件                               | ✅                         |
| 全局状态         | Zustand（`projectStore`、`canvasUiStore` 等）               | ✅                         |
| DAG 执行引擎     | Rust `graph` + `executor::run_graph`                    | ⚠️ **有骨架，缺完整状态机/并行/断点续跑** |
| 资产与协议        | Rust `db`、素材索引与导入返回 `asset_id`、前端画廊/列表对齐；**执行器尚未只认 ID** | ⚠️                        |
| 算子内核         | Rust 部分能力 + 前端 `invoke` 分散调用                            | ⚠️                        |
| 渲染与编码        | 预览偏 Web；FFmpeg 在设置与拼接等中使用                               | ⚠️                        |

## 六、结论与后续迭代建议（非规范，仅便于排期）

- **设计文档**指向：**强类型端口 + 统一资产与元数据 + 完整 DAG 执行语义（状态、失败策略、并行、续跑）+ 脚本驱动的时序与镜头 ID**。
- **当前仓库**更接近：**React Flow 画布 + 宽松节点数据 + DAG 校验与串行执行雏形 + 若干独立生成/拼接能力**；与文档在 **执行模型、数据协议、脚本中枢** 上仍有系统性差距。
- 迭代时可优先选一条纵轴打穿（例如：**资产 ID + 元数据表 + 执行器只认 ID**），再扩展端口类型与节点状态机，避免长期停留在表现层。
